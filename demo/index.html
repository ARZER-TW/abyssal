我们在解决什么问题

隐私在密码学中有两种截然不同的含义，长期以来被混为一谈。

数据隐私 (Data Privacy)

计算逻辑公开，输入数据隐藏。

Aztec、Aleo、ZK Rollup 正在解决这个问题。已有成熟方案。

私有函数评估 (Private Function Evaluation, PFE)

计算逻辑本身也是隐藏的，但结果仍可信验证。

学术界讨论了几十年，从未有 Web3 生产级实现。

传统金融机构的核心竞争力在于算法——信用评分模型、风控引擎、合规逻辑。这些是数十年积累的商业机密。它们想进入 DeFi，却面临一个无解的困境：

选择一：把算法放上公链
  -> 逻辑完全公开，竞争对手一天内复制
  -> 机构会拒绝

选择二：维持中心化服务
  -> 用户必须盲目信任机构
  -> 和 Web2 没有本质区别

这个困境在今天没有解法。现实世界用「可信中间人」（麦肯锡、评级机构、审计所）来绕开它，每年市场规模数千亿美元。

Abyssal 是第一个让「算法私密、结果可信、无需盲目信任中间人」同时成立的 Web3 协议。

核心洞察：一个被忽视的密码学事实

Groth16 Trusted Setup 产生两把性质完全不同的密钥。

Proving Key (pk)

大小5-50 MB
内容电路逻辑的完整多项式编码
风险有了它可以分析出电路在做什么
处理必须加密保护

Verification Key (vk / pvk)

大小~700 bytes (4 byte vectors)
能力验证 proof 是否合法
安全从数学上完全无法反推电路逻辑
处理公开部署上链
链上验证一个 proof，只需要 vk，完全不需要 pk。

在传统 ZK 应用中，用户需要在自己的设备上生成 proof，因此 pk 必须公开分发。Abyssal 改变了这个架构前提：将 proof 生成移到 TEE（可信执行环境）内部，用户不再需要直接接触 pk，因此 pk 可以被加密保护。

同时，wasm（由电路编译产生的执行档，包含完整的计算步骤，懂逆向工程的人可以从中还原出算法逻辑）也必须一并加密。pk 和 wasm 都被 Seal IBE 加密后存储在 Walrus 上，只有经过授权的 TEE 才能解密和使用。vk 正常上链，任何人可以验证 proof 是否合法，但无法推断电路在验证什么。

这在 Web3 产品中尚无先例。

// Deploy Phase — 业务逻辑所有者执行一次

银行把自己的信用评分算法「封装」成一个链上服务。算法本身被加密保护，链上只留下一个验证接口（vk）。任何人都能用这个接口验证结果，但没有人能看到算法在做什么。

1
写算法 — 把信用评分逻辑写成 Circom 电路（ZK 电路语言）
2
生成密钥对 — Trusted Setup 产出 pk（5-50 MB，包含算法逻辑）和 vk（~700 bytes，只能验证）
3
加密 + 存储 — pk + wasm 用 Seal IBE 加密，存入 Walrus 去中心化存储。只有合法 TEE 能解密
4
上链 — vk 部署到 Sui Move 合约。此时链上只有 ~700 bytes，任何人看不出这在验证什么

VaultConfig — 链上存储的内容

// Move 合约中 VaultConfig 的核心字段
public struct VaultConfig has key {
    id: UID,
    owner: address,

    // Groth16 验证密钥 (~700 bytes)，只能验证、无法反推电路逻辑
    pvk: groth16::PreparedVerifyingKey,

    // 电路身份指纹：SHA-256(SHA-256(wasm))，32 bytes
    wasm_double_hash: vector<u8>,

    // 加密文件在 Walrus 上的位置（内容寻址，篡改即失效）
    pk_blob_id: vector<u8>,
    wasm_blob_id: vector<u8>,
    circuit_source_blob_id: vector<u8>,

    // 访问控制策略（Seal policy 合约 ID）
    seal_policy_id: ID,

    // 已注册的 TEE 公钥列表（只有这些 TEE 能解密 pk/wasm）
    registered_enclave_pubkeys: vector<vector<u8>>,

    // Proof 有效期（28 个 Sui epoch ≈ 28 天）
    proof_validity_epochs: u64,

    // 防重放策略：0=一次性, 1=每 epoch 一次, 2=无限制
    nullifier_policy: u8,
}

// User Phase — 每次计算

用户把隐私数据交给一台「谁都偷看不了的机器」(TEE)，机器用加密的算法算出结果，输出一个数学证明。全程没有人看到数据，也没有人看到算法。

1
提交数据 — 用户把收入、支出、信用历史等私有数据发送给 TEE（可信执行环境，硬件保证管理员也看不到内容）
2
获取密钥 — TEE 向 Seal 证明「我是合法的、没被篡改的机器」，Seal 验证通过后释放 pk/wasm 的解密密钥
3
密封计算 — TEE 在硬件安全边界内：解密 wasm 处理用户数据（得到信用分 650）→ 解密 pk 生成 Groth16 proof（数学证明「650 是按正确规则算出来的」）
4
输出凭证 — 128 bytes proof + 4 个 Poseidon hash（看不出在证明什么）→ 打包成 VaultProof Sui 物件，归用户所有

// Verification Phase — 任意次数

用户拿着 proof 去任何 DeFi 协议。协议只需要在链上做一次数学验证（极低成本），不需要知道算法是什么，不需要联系银行，不需要重新计算。

1
提交 proof — 第三方协议接受用户的 VaultProof Sui 物件
2
链上验证 — 调用 sui::groth16::verify_groth16_proof(pvk, proof, public_inputs)，固定成本（2 次 BN254 配对运算）
3
执行业务 — 验证通过 → 放款、授权、解锁等业务逻辑自动执行
4
复用 — 同一个 proof 物件可在有效期内被多个不同协议重复验证使用，不需要重新计算

链上数据会不会泄露算法？

Proof 和 public inputs 都在链上公开可见。一个自然的疑问：观察者能不能通过分析这些数据，反推出算法在做什么？

答案是：不能——前提是电路的输出设计遵循了正确的规范。我们要求所有部署到 Abyssal 的电路，必须用 Poseidon hash 将业务语义完全封装，链上只暴露 4 个不透明的哈希值。观察者能看到这些值，但无法从中推断出任何业务含义。

错误设计 -- 泄露电路语义

// 任何人都能看出这是什么系统
min_credit_score: 700     <- 泄露：在看信用分
approved: bool            <- 泄露：审批系统
income_verified: bool     <- 泄露：在验证收入

正确设计 -- Poseidon hash 封装

// 只有 Poseidon hash + epoch
nullifier: Poseidon(secret, vault_id, epoch)
result_commitment: Poseidon(result, salt)
vault_id_hash: Poseidon(vault_id)
expiry_epoch: 128

非授权观察者只知道：「某用户对某 vault 生成了一个有效 proof」。不知道 proof 在证明什么。

credit_score.circom — 实际电路代码

// 这就是被 Seal 加密保护的「秘密公式」
// 观察者看不到这段代码，只能看到 4 个 Poseidon hash 输出

template CreditScore(VALIDITY_DURATION) {
    // ===== 标准私有输入 =====
    signal input user_secret;
    signal input vault_id_field;
    signal input epoch;
    signal input result_value;
    signal input result_salt;

    // ===== 业务逻辑私有输入（这部分是「秘密」）=====
    signal input income;
    signal input monthly_expenses;
    signal input years_of_history;

    // ===== 秘密公式：score = 300 + 净收入 + 年资*10 =====
    signal computed_score;
    computed_score <== 300 + income - monthly_expenses
                       + years_of_history * 10;
    result_value === computed_score;  // 绑定：结果必须等于公式算出的值

    // ===== 4 个语义不透明的输出（Poseidon hash）=====
    signal output nullifier;           // Poseidon(secret, vault_id, epoch)
    signal output result_commitment;   // Poseidon(result, salt)
    signal output vault_id_hash;       // Poseidon(vault_id)
    signal output expiry_epoch;        // epoch + 28
    // ... Poseidon 计算逻辑省略 ...
}

指定审计方的完整验证链

算法虽然对公众隐藏，但部署方可以选择让指定人士（监管机构、投资方、审计员）完整审计电路逻辑，而不需要对外公开。审计方通过以下五层独立验证，不需要相信部署方的任何口头承诺。

Layer 1: 电路身份确认

审计员通过 Seal 解密拿到电路原始码（.circom），亲眼看到算法逻辑：「score = 300 + income - expenses + years * 10」。这不是部署方口头告诉他的，是他自己从链上加密存储中解密出来的。

技术：Seal AuditAllowlistPolicy 解密 -> 获得 circuit.circom (3,420 bytes)

Layer 2: 电路与链上绑定确认

审计员拿到原始码后，自己重新编译，计算出一个指纹（hash）。然后和链上 VaultConfig 里存的指纹比对——如果完全一致，就数学证明了「链上这个 vault 用的确实是我手上这份电路，不是别的东西」。

技术：重新编译 .circom -> SHA-256(SHA-256(wasm)) -> 和链上 wasm_double_hash 比对

Layer 3: TEE 自报告确认

TEE 在生成 proof 时，会把「我用的是哪个 wasm」的指纹写进硬件证明文件。审计员比对这个指纹，确认 TEE 内部确实在用正确的电路，而不是偷偷换了一个。

技术：TEE attestation user_data 包含 H(H(wasm))，和链上比对

Layer 4: PCR 锁定确认

审计员拿到 TEE 的程序源代码，自己编译一遍，算出一个「代码指纹」(PCR)。和链上白名单里的 PCR 比对——确认 TEE 跑的代码没有被篡改，Layer 3 的自报告是可信的。

技术：独立编译 TEE 代码 -> 计算 PCR0/1/2 (SHA-384) -> 和 Seal policy 白名单比对

Layer 5: Trusted Setup Ceremony 参与

在最高安全需求的场景，审计员亲自参与密钥生成仪式。只要参与者中至少一个人是诚实的，整个系统就是安全的。这让审计员从「被告知电路是什么」变成「亲自参与确认电路是什么」。

经过以上验证，你不再需要信任：

  • 不需相信机构没偷偷换规则 — vk 固定上链，每次 proof 都对应同一套逻辑，数学保证
  • 不需相信机构对所有用户用同样标准 — 同一个 vk，数学上完全一致，电路结构上不可能存在歧视性分支

Seal — 去中心化密钥访问控制

用 Move 合约决定谁能解密，而不是中心化服务器

把「谁有资格解密 pk/wasm」的答案，从中心化服务器的 ACL 变成 Move 合约逻辑。通过 IBE（基于身份的加密）+ DKS MPC 委员会实现去中心化，无单一节点持有完整密钥。

Abyssal 用了两个 Seal policy：VaultKeyPolicy（TEE 解密 pk/wasm）+ AuditAllowlistPolicy（审计员解密电路源码）

// seal_approve — Seal key server 通过 dry-run 执行此函数
// 所有 7 个条件必须全部通过，才会释放解密密钥

Walrus — 去中心化加密存储

让大型加密文件有去中心化、防篡改的存储

为 5-50 MB 的加密 pk/wasm 提供去中心化、内容寻址的存储。blob ID = H(content)，文件被替换则 ID 改变，任何人可独立验证完整性。blob 本身是 Sui 链上物件，可被 Move 合约引用、设定访问控制、管理生命周期。

Abyssal 存了 3 个加密 blob：pk + wasm + 电路源码，blob ID 记录在链上 VaultConfig 中

Nautilus — 可验证链外计算

在硬件安全边界内执行计算，算法和数据从不暴露

在 AWS Nitro Enclave 硬件安全边界内执行 proof 生成。wasm 和用户数据从不离开 enclave。PCR attestation 把「计算发生在可信环境」从口头承诺变成可被 Move 合约验证的密码学哈希。最关键的安全分层：TEE 被攻破只导致算法泄露，资产安全不受影响（底线是链上 vk 的数学保证）。

对比纯 TEE 隐私链：TEE 被攻破 = 用户资产直接被盗。Abyssal：TEE 被攻破 = 部署方算法泄露（商业损失），但用户链上资产绝对安全

Sui Move + Groth16 — 链上验证与物件模型

固定成本验证任意复杂的 ZK proof，proof 作为 Sui 物件跨协议流通

sui::groth16 是 Sui 框架原生 API，验证成本恒定（2 次 BN254 配对运算），不论电路有多复杂。Sui 物件模型让 VaultProof 成为第一类公民——带有所有者、有效期、nullifier，可以在整个 Sui 生态的任意协议里验证使用，像代币一样流通。

23 个 Move 单元测试，9 步链上验证，proof 作为可跨协议复用的 Sui 物件

// submit_proof — 9 步链上验证，全部通过才创建 VaultProof
public fun submit_proof(vault, enclave_pubkey, proof, inputs, sig, ctx) {
    // 1. Vault 未暂停
    assert!(!vault.paused);
    // 2. TEE 公钥已注册到此 Vault
    assert!(is_enclave_registered(vault, &enclave_pubkey));
    // 3. Ed25519 签名验证（TEE 签了 proof||inputs）
    assert!(ed25519::verify(&sig, &enclave_pubkey, &message));
    // 4. Public inputs 长度 = 128 bytes（4 x 32）
    assert!(length(&inputs) == 128);
    // 5. vault_id_hash = Poseidon(vault_id % BN254_R)
    assert!(vault_id_hash == expected_hash);
    // 6. Epoch 有效期检查
    assert!(current_epoch <= expiry_epoch);
    // 7. Nullifier 防重放
    assert!(!table::contains(&vault.used_nullifiers, nullifier));
    // 8. Groth16 proof 验证（核心：2 次 BN254 配对）
    assert!(groth16::verify_groth16_proof(&bn254(), &vault.pvk, &inputs, &proof));
    // 9. 创建 VaultProof Sui 物件，归用户所有
    VaultProof { vault_id, nullifier, result_commitment, ... }
}

和现有方案的根本差距

vs Aleo / Aztec (ZK 隐私网络)

他们解决

Data Privacy -- 数据隐藏，逻辑公开

Abyssal 解决

Private Function Evaluation -- 逻辑本身隐藏

完全不同的问题。他们的架构设计上就不是为了隐藏合约逻辑。

vs Secret Network / Phala / Oasis (全 TEE 隐私链)

他们的做法

整个智能合约塞进 TEE

致命缺陷：无链上密码学验证。TEE 被攻破 -> 伪造结果 -> 窃取资产

Abyssal 的做法

TEE 只生成 proof

资产安全底线 = 链上 vk（数学，不是硬件）

这是安全模型层面的质的差异，不是程度差异。

vs FHE / zkVM (RiscZero, SP1)

他们的目标

纯数学保护逻辑

现实 (2026)：FHE 比明文慢 100-1000x，zkVM proof 生成需要分钟级

Abyssal 的平衡

Groth16 + TEE = 实用平衡

TEE proof 5-60s，链上验证固定低成本

在效率和隐私之间找到目前技术下唯一合理的商业平衡点。
